【峽谷行者】族語唸錯的那一題：一個「隨機化」留下的靜音陷阱

摘要

我在做一款太魯閣族語的卡牌遊戲，出牌前要答一題族語，答完會自動唸出正解幫你記發音。有天玩家說「族語都唸錯了、聽起來像上一題的答案」。我第一時間以為是音檔快取或播放競態——結果都不是，真正的兇手是我自己幾週前一個「把題目跟卡片脫鉤」的改版，只改了一半。

改了好多天結果題目跟答案不一樣

訊息很短：

「我剛剛進去玩單機版，題目的族語都唸錯了欸，聽起來都是上一題的答案。」

這個遊戲的核心賣點就是「邊玩邊學族語」——答完題、揭曉正解時，會自動播一次那個詞的官方發音，做視聽同步。如果發音是錯的，那學習這件事直接崩掉，比畫面醜還嚴重。所以我立刻停下手邊的事去查。

而且「聽起來像上一題」這句話，聽起來太像某一類經典 bug 了——這正是它騙到我的地方。

我原本以為問題在哪

「像上一題的答案」，在我腦裡直接對應到兩個嫌疑犯：

  1. 音檔快取沒清：播放器沿用了上一輪載入的音檔。
  2. 非同步競態（async race）：這一題的播放請求，被上一題還沒結束的請求蓋掉了。

這兩個都會製造出「慢半拍、放到舊的那個」的感覺。我本來已經準備往「幫播放加上請求版本號、把舊的取消掉」的方向修了。

後來我發現真正的問題

我先去讀出題和播放的程式碼，而不是急著改。然後就看到了真相——跟「上一題」一點關係都沒有。

背景是這樣：這遊戲早期每張卡綁定一個固定的族語詞，出牌就考那個詞。後來為了「之後擴充卡池時不用逐張配詞」，我把出題改成每次從整個詞庫隨機抽一個詞。從那刻起，題目問的詞，跟卡片本身綁的詞，就是兩回事了。

但是——播放發音的那一行，沒跟著改：

const audio = quiz.audioUrl ?? audioUrl(quiz.card.vocabId);

單字題的 quiz.audioUrl 是空的，於是它 fallback 去查 quiz.card.vocabId——卡片自己綁的那個固定詞的發音。

所以玩家聽到的根本不是「上一題」，而是「這張卡永遠固定綁的那個詞」。因為題目是從一千多個詞裡隨機抽的，它跟卡片綁的詞幾乎永遠不一樣，於是每一題都放錯。之所以感覺像「上一題」，只是因為它一直是個不相干的、固定的聲音——大腦把「不對」腦補成了「好像剛剛聽過」。

真正的一句話根因是：

我把「題目」跟「卡片」脫鉤時，只改了「出題」，忘了改「消費出題結果的另一端」（播放發音）。舊的耦合以一個 ?? fallback 的形式殘留下來，靜靜地放錯音。

我怎麼處理

修法其實很小，難的是找到它。

需要準備什麼／工具

  • 專案本身（Next.js + TypeScript）、一個能跑的開發伺服器。
  • 不需要額外訂閱或 API：族語發音檔是既有的靜態資源（詞庫每個詞條自帶一個發音 URL）。
  • 驗證時用到瀏覽器的開發者主控台，這樣就夠了。

實際步驟

  1. 讓出題函式在產生題目的當下，就把「這次抽到的詞」的發音 URL 一起塞進題目狀態裡（跟困難模式的句子題共用同一個欄位、同一條播放路徑）。
  2. 播放端只信任這個「題目自帶的發音」，把查卡片詞的 fallback 整行刪掉——因為那條路只會放錯音。
  3. 順手把型別註解改清楚：這個欄位代表「這一題實際問的詞的發音」，不是卡片的詞，免得下一個人（或下一個我）再踩。

遇到的坑

  • 最大的坑是那個 ?? fallback。它讓程式「看起來有處理」，type check 也過、build 也綠、畫面也不會壞——它只是安靜地放錯聲音。能跑不代表對。
  • 誤導性的症狀（「像上一題」）差點讓我修錯地方。如果我照著症狀去加「取消上一個請求」的機制，bug 一樣還在，而且我還多寫了一堆沒用的複雜度。

怎麼驗證有沒有成功 我沒有只用耳朵，也沒有只相信自己讀的程式碼。我在瀏覽器主控台把 window.Audio 這個建構子攔截（monkey-patch）起來，記錄下實際被拿去播放的音檔網址，然後三方對照：

詞音檔
畫面問的題毒（正解 sahu）—
修好後實際播放sahu／毒.../32-42.wav ✅ 對上了
這張卡自己綁的詞cimu／鹽.../21-07.wav ← 修好前就是誤播這個

播放的音檔跟題目對上、而且明確跟「卡片綁的舊詞」不同——這才算真的修好，不是「我覺得應該好了」。

這次真正花掉的是什麼

程式改動只有三個檔案、加起來沒幾行。真正花掉的是兩段判斷力：

  • 忍住不照症狀直接開修。「像上一題」很有畫面感，但它是誤導。多花十分鐘讀程式碼，比少花十分鐘寫錯的修法划算太多。
  • 不相信「看起來合理」。這個 bug 全程 type check 綠、build 綠、UI 正常，只有聲音錯。如果我沒有真的去攔截播放網址，我很可能改完就以為好了。

如果你也想做一次

給你一個馬上能用的小起點：當你懷疑「播出來/送出去的東西不對」，別用眼睛猜，去攔截真正發生的那一步。

以網頁音訊為例，在主控台貼這段，就能看到實際被播放的每一個音檔網址：

const Orig = window.Audio;
window.Audio = function (url) { console.log("play:", url); return new Orig(url); };

然後去操作一次，比對「畫面上該播的」跟「主控台印出來的」是否一致。這招不只音訊，表單送出（攔 fetch）、埋點事件、下載連結，都適用——驗證的是「真的發生了什麼」，不是「程式碼看起來會做什麼」。

我這次帶走的做法

整理成三條，之後每次都能複用：

  1. 脫鉤要成對地改。 當你把 A 和 B 解除綁定（例如題目不再綁卡片），用搜尋把「所有還在讀舊關聯的地方」全部找出來，一次改乾淨。漏掉的那一端就是下一個 bug。
  2. 警惕 `a ?? 舊路徑` 這種 fallback。 它常常是重構沒清乾淨的殘骸——不會報錯，只會在你沒注意時，悄悄走回那條你以為已經廢掉的舊路。
  3. 驗證要看「實際輸出」，不是「程式碼意圖」。 攔截真正被執行的那一步（播放、請求、寫入），拿它跟預期對照。綠燈不是通過，對得上才是。
